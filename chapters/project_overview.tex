\section{About This Project}

\subsection{Project Description}

This report documents the development of a computer simulation of the classic board game Clue! (Cluedo) for the fictional client Watson Games. The system was developed in Unity (C\#) over four Agile sprint cycles between February and April.

The final deliverable is a desktop game for Mac/PC that supports multiple players (human or AI), featuring:
\begin{itemize}
    \item A faithful 25-row $\times$ 24-column tile representation of the Clue! board with all 9 rooms, corridors, doors, starting squares and secret passages
    \item Physics-based dice rolling (Rigidbody simulation with a fallback for degenerate results)
    \item Orthogonal token movement along corridors, bounded by the dice roll, with room entry detection
    \item A 21-card deck (6 characters, 6 weapons, 9 rooms) dealt from a 3-card murder envelope, with private per-player hand display
    \item Suggestion system: clockwise card-showing round with private card reveal via modal, and suggested token movement into the suggestion room
    \item Accusation evaluation against the murder envelope, with win condition and player elimination
    \item An AI player that takes full turns (roll, move, suggest, optionally accuse) without human input using BFS and basic knowledge tracking
    \item Per-player detective notepad (147-cell toggle grid)
    \item Game-over screen with restart and return-to-menu options
    \item Data-driven card names loaded from \texttt{cards.json} --- characters, weapons and rooms are configurable without recompiling
\end{itemize}

\section{How the Team Was Organised}

Roles were assigned by area of responsibility at Meeting 3 (19/02/2026), with each team member owning one or two scripts per sprint. Matt handled frontend and Unity rendering. Ewan owned the core game logic and turn management. Joe and Ethan shared board and movement responsibilities. Ben owned the card system, dice, and AI player. Edward owned data handling, JSON integration, and repository tooling.

Code review was conducted via GitHub Pull Requests. Edward set up a PR template and a Unity compile-check workflow to prevent broken builds from reaching \texttt{master}. Each team member documented their own commits in the agreed schematic format.

\subsection{Technology Choices}
\begin{itemize}
    \item Unity / C\#: Agreed unanimously at Meeting 1. The team had little to no Unity experience and saw it as a valuable learning opportunity. Unity provides built-in 3D rendering, Rigidbody physics for the dice, and cross-platform Mac/PC builds.
    \item Backend/frontend architecture: Agreed at Meeting 1. The backend outputs game state, and Matt's frontend consumes it. \texttt{GameManagerScript} serves as the single bridge, with UI scripts never reading board or deck state directly. This approach kept game logic testable independently of Unity's rendering pipeline.
    \item Data-driven design: All card names (characters, weapons, and rooms) are loaded from \texttt{StreamingAssets/cards.json} via \texttt{JsonUtility}. Agreed at Meeting 3, this satisfies Watson Games' customisation requirement. Changing a character or room name requires only a JSON edit, with no recompile needed.
    \item C\# XML doc comments: Agreed at Meeting 3 as the documentation standard. This provides IDE-surfaced hover tooltips and is compatible with DocFX for HTML generation.
\end{itemize}

\subsection{Version Control}
Git and GitHub were used throughout the project. The branching strategy followed feature branches per task, then PR review, followed by merge to \texttt{master}. Git LFS was configured in Sprint 2 for Unity binary files. Unity Smart Merge with \texttt{.gitattributes} for \texttt{.unity} and \texttt{.prefab} files was set up by Edward in Sprint 2 to prevent destructive scene-file merge conflicts.

\section{Key Achievements}

\begin{itemize}
    \item Board: A 25 by 24 tile grid (\texttt{BoardScript}) encodes all 9 rooms (tile values 2 to 11), corridors (value 1), walls (value 0), door squares, and character starting squares. The room layout faithfully reproduces the standard Cluedo board.
    \item Physics dice: \texttt{DiceRollScript} uses a Unity Rigidbody simulation rather than \texttt{Random.Range}, giving the dice a visible roll animation. Face orientation is decoded by dot product against \texttt{Vector3.up}. A \texttt{Random.Range(1,7)} fallback handles any degenerate result, and risk R5 (dice giving zero results) never triggered in testing.
    \item Movement: Orthogonal token movement bounded by the dice roll value, with occupancy tracking so that two tokens cannot share a corridor square. The implementation includes room-entry detection and secret-passage support.
    \item Card system: A 21-card deck (6 characters, 6 weapons, 9 rooms) is loaded from \texttt{cards.json}. Three cards (one of each type) are selected as the murder envelope before dealing, and the remaining 18 are dealt round-robin. Each player's hand is displayed in a private UI visible only to that player.
    \item Suggestion and accusation: The complete suggestion flow allows the suggesting player to name their character and weapon, with the room fixed as the current room. The suggested character token is moved into the room. A clockwise card-showing round follows, with each player prompted to reveal one matching card privately via \texttt{UiManager.ShowRevealedCard}, ending when the first match is shown. Accusations are evaluated against \texttt{Deck.GetEnvelope()}: a correct accusation triggers the win screen, while an incorrect one eliminates the player from future turns while keeping their hand available for responses.
    \item AI player: \texttt{AiPlayer} takes full turns without human input, rolling dice, selecting a good legal move (BFS-directed toward envelope-candidate rooms), entering rooms, and making a suggestion from the character and weapon cards not yet eliminated from its knowledge base. It accuses when envelope candidates narrow to exactly 3, with each turn completing in under 5 seconds.
    \item Detective notepad: A 147-cell toggle grid (\texttt{NotepadGrid} and \texttt{NotepadUI.cs}) is available per human player. It persists across turns and allows players to manually track which cards have been shown.
    \item Process and engineering: Nine documented team meetings with agreed actions and completion tracking. Four sprint cycles each included retrospectives. A PR-driven workflow with Unity compile checks on every PR prevented broken builds from landing on \texttt{master}. Git LFS and UnityYAMLMerge prevented destructive scene-file conflicts after Sprint 2.
\end{itemize}

\section{What Went Well}

\begin{itemize}
    \item Repository tooling prevented merge disasters. Setting up Git LFS, \texttt{.gitattributes} for Unity Smart Merge, and PR templates at the start of Sprint 2 eliminated Unity scene and prefab merge conflicts for the rest of the project. Risk R2 materialised in Sprint 2 but was immediately contained and did not recur. Without this tooling, late-sprint parallel feature work would have been extremely fragile.
    \item Parallel development by area kept work non-blocking. Assigning ownership of one or two scripts per person per sprint (agreed at Meeting 3) meant team members rarely blocked each other. The card system (Ben), board rendering (Matt), game manager (Ewan), and movement (Joe and Ethan) all progressed in parallel across Sprints 2 and 3 with minimal dependency conflicts.
    \item Architecture decisions made early paid off throughout. The backend/frontend separation (Meeting 1), the \texttt{GameManagerScript} singleton as the single bridge (Meeting 5), and the data-driven \texttt{cards.json} approach (Meeting 3) were all agreed before coding began. These decisions kept each area independently developable and allowed Watson Games' customisation requirement to be satisfied in the future without rework.
    \item End-to-end delivery of Sprint 4 critical path. Every critical-path objective for Sprint 4 was completed: full turn cycle, clockwise suggestion resolution, accusation evaluation, AI player, and game-over screen. The late-sprint integration push consolidated all major feature branches without losing work.
    \item Honest documentation of limitations. Rather than leaving known bugs undocumented, the team explicitly logged known issues (KI-1 through KI-3) in this report.
    \item Risk R5 never materialised. The physics dice fallback was planned defensively from Sprint 1. The dice never returned an invalid result despite extensive testing.
    \item Risk R6 presented a minor issue, but didn't end up causing many a big delay. When it became clear it was going to take a long time to do A* and full deduction, the team was quick to simplify the goal to mitigate time loss.
\end{itemize}

\section{What Didn't Go Well / Lessons Learned}

\begin{itemize}
    \item Easter break was not budgeted into the plan. The 3.5-week Easter pause (20 March to 14 April) was completely foreseeable but not reflected in the Gantt chart or sprint dates. Sprint 3 work was compressed into 15 to 24 April, overlapping heavily with Sprint 4 and creating significant time pressure. Lesson: cost all known vacation windows into the plan from day one.
    \item Sprint 2 lost approximately one week to unplanned repository tooling. Git LFS setup, \texttt{.gitignore} fixes, PR templates, and Unity merge settings were all necessary but should have been done in Sprint 1 before any feature work began. Lesson: front-load all toolchain work on day one of Sprint 1, before feature branches open.
    \item TokenMovement was harder than estimated and slipped. The planned end date was 15 April, but significant movement fixes landed on 17 to 18 April. Legal-move detection on the irregular Cluedo board required substantially more edge-case handling than anticipated, and the task blocked development for everything downstream. Lesson: for the longest single task, add an explicit buffer and consider stubbing the interface earlier so other work can proceed.
    \item Late integration push created architectural debt. Because the game loop, suggestion, accusation, and AI player all converged in the final 10 days, the Suggestion and Accusation classes could not be extracted cleanly. They were merged into \texttt{GameManagerScript} and \texttt{UiManager} under time pressure (KI-2). Lesson: stub the end-to-end game loop in Sprint 1, even if every action just logs to the console. Building outwards from a working skeleton would have surfaced integration problems much earlier.
    \item Documentation was partly reconstructed rather than maintained in real time. Several sprint docs were filled in retrospectively during the Sprint 4 documentation push rather than being written as each sprint ran. Lesson: designate one person per sprint to maintain the sprint doc live, not retroactively.
    \item Two of the final five meetings were held without Matt. Some Sprint 4 UI integration decisions were made without Matt's direct input and required follow-up via async communication, adding friction at a critical time. Lesson: agree a minimum-attendance rule early. If a member cannot attend, ensure decisions are communicated to them within 24 hours with an explicit confirmation required.
    \item Sprint 4 system tests were not fully executed before submission. The system test table (S4.1 to S4.12) has complete test cases and expected outputs, but the Actual Result and Pass/Fail columns could not be filled in before the deadline. This is the most significant gap against the 9 to 10 testing band, which requires full documentation including pass/fail verdicts.
    \item AI player does not reach full deduction sophistication. The \texttt{AiPlayer} uses BFS-directed navigation toward envelope-candidate rooms and filters suggestions against its \texttt{PlayerKnowledge} base, but does not perform full card-elimination deduction (KI-1).
    \item Watson Games' customisation requirement was only partially met. The team delivered four selectable board themes, but player names are not user-configurable. Weapon and character art packs loadable from data files were identified as future work but not delivered.
    \item No animated visual appearance. The physics-based dice provides some animation, but character tokens do not animate on movement.
\end{itemize}

\section{What We Would Do Differently}

\begin{itemize}
    \item Stub the end-to-end game loop in Sprint 1. Even if every action is a \texttt{Debug.Log}, a working turn cycle (roll, move, suggest, accuse, and win check) from the very beginning would have given every other system a defined interface to plug into. Token movement, suggestion, and accusation were all urgent in Sprint 4 precisely because the loop did not exist earlier.
    \item Treat the Easter break as a hard Sprint 3 deadline, not a mid-sprint pause. Sprint 3 should have closed its deliverables before 20 March. Planning the sprint to finish before Easter would have preserved Sprint 4 as a genuine testing and integration buffer rather than a compressed feature sprint.
    \item Set up all repository tooling on day one of Sprint 1. Git LFS, \texttt{.gitattributes}, Unity Smart Merge, PR templates, and the compile-check workflow should all be done before the first feature branch opens. Toolchain failures early in a sprint cascade into feature delays.
    \item Allocate one person per sprint to maintain documentation alongside development. A rotating ``sprint scribe'' role would have prevented the retroactive reconstruction near the deadline. Someone responsible for keeping the sprint doc current throughout the sprint would have been more efficient than updating everything at the end.
    \item Define a minimum viable AI contract early, then expand. The random-decision AI was the right minimum, and the architecture (\texttt{PlayerKnowledge}, abstract decision hooks on \texttt{Player}) was well designed for layering on deduction. However, this design should have been locked in Sprint 1 and expanded incrementally, rather than being finalised under Sprint 4 time pressure with deduction left unimplemented.
\end{itemize}

\section{Ideas for Further Development}

\begin{itemize}
    \item A smarter AI player using card-elimination deduction logic (the \texttt{PlayerKnowledge} scaffold is already in place --- \texttt{GetEnvelopeCandidates()} is implemented and wired into the AI)
    \item Animated token movement along the path (rather than step-by-step per key press)
    \item Online/networked multiplayer support (the abstract \texttt{Player} base class with decision hooks would accommodate a \texttt{NetworkPlayer} subclass without changes to \texttt{GameManagerScript})
    \item Custom board, character and weapon packs loaded from Watson Games data files (foundation already in place via \texttt{cards.json})
    \item Mobile port (Unity's cross-platform build targets make this achievable; touch input would need to replace keyboard movement)
\end{itemize}
